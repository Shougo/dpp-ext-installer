==============================================================================
INTRODUCTION                                  *dpp-ext-installer-introduction*

This ext implements plugins installer.

NOTE: It does not work for |dpp-plugin-option-frozen| and
|dpp-plugin-option-local| plugins.


==============================================================================
INSTALL                                            *dpp-ext-installer-install*

Please install both "dpp.vim" and "denops.vim".

https://github.com/Shougo/dpp.vim
https://github.com/vim-denops/denops.vim

And you need to install protocol like "dpp-protocol-git".

https://github.com/Shougo/dpp-protocol-git

ACTIONS                                            *dpp-ext-installer-actions*

                                              *dpp-ext-installer-action-build*
build
		Build plugins.
		NOTE: It is also executed when plugins are installed/updated.

		params:
			{names}:	Plugin names.
					(Default: [])

                                    *dpp-ext-installer-action-checkNotUpdated*
checkNotUpdated
		Check not updated or not installed plugins via the protocol.
		It does not return value but prompts whether update.

		params:
			{names}:	Plugin names.
					(Default: [])
			{force}:	Force update plugins if possible.
					(Default: false)

                                          *dpp-ext-installer-action-denoCache*
denoCache
		Execute "deno cache --reload" for plugins.
		NOTE: "deno" command is required.
		NOTE: It is also executed when plugins are installed/updated
		if |dpp-ext-installer-param-enableDenoCache| is v:true.

		params:
			{names}:	Plugin names.
					(Default: [])

                                          *dpp-ext-installer-action-getFailed*
getFailed
		Get update failed plugins from previous update.
		NOTE: It must be executed by |dpp#sync_ext_action()|.

                                            *dpp-ext-installer-action-getLogs*
getLogs
		Get all previous logs.
		NOTE: It must be executed by |dpp#sync_ext_action()|.

                                    *dpp-ext-installer-action-getNotInstalled*
getNotInstalled
		Get not installed plugins.
		NOTE: It must be executed by |dpp#sync_ext_action()|.

		params:
			{names}:	Plugin names.
					(Default: [])

                                      *dpp-ext-installer-action-getNotUpdated*
getNotUpdated
		Get not updated plugins via the protocol
		(same as |dpp-ext-installer-action-getUpdated|).
		NOTE: It must be executed by |dpp#sync_ext_action()|.

		params:
			{names}:	Plugin names.
					(Default: [])

                                         *dpp-ext-installer-action-getUpdated*
getUpdated
		Get updated plugins from previous update.
		NOTE: It must be executed by |dpp#sync_ext_action()|.

                                      *dpp-ext-installer-action-getUpdateLogs*
getUpdateLogs
		Get all previous update logs.
		NOTE: It must be executed by |dpp#sync_ext_action()|.

                                            *dpp-ext-installer-action-install*
install
		Install plugins.

		NOTE: Before installing, the current plugin revisions are
		saved to a rollback snapshot under
		"{base-path}/{name}/rollbacks/{YYMMDDhhmmss}/rollback.json".
		The special directory "latest" is also updated to always
		point to the most recent snapshot.

		params:
			{names}:	Plugin names.
					(Default: [])
			{rollback}:	Restore plugins to a previously saved
					revision snapshot instead of installing
					fresh.  Use "YYMMDDhhmmss" to target a
					specific snapshot, or "latest" to
					restore the state from the most recent
					snapshot.
					(Default: "")

                                          *dpp-ext-installer-action-reinstall*
reinstall
		Reinstall plugins.

		NOTE: Before reinstalling, the current plugin revisions are
		saved to a rollback snapshot under
		"{base-path}/{name}/rollbacks/{YYMMDDhhmmss}/rollback.json".
		The special directory "latest" is also updated to always
		point to the most recent snapshot.

		params:
			{names}:	Plugin names.
					(Required)
			{rollback}:	Restore plugins to a previously saved
					revision snapshot instead of
					reinstalling fresh.  Use "YYMMDDhhmmss"
					to target a specific snapshot, or
					"latest" to restore the state from the
					most recent snapshot.
					(Default: "")

                                             *dpp-ext-installer-action-update*
update
		Update plugins.

		NOTE: Before updating, the current plugin revisions are
		saved to a rollback snapshot under
		"{base-path}/{name}/rollbacks/{YYMMDDhhmmss}/rollback.json".
		The special directory "latest" is also updated to always
		point to the most recent snapshot.

		params:
			{names}:	Plugin names.
					(Default: [])
			{rollback}:	Restore plugins to a previously saved
					revision snapshot instead of updating
					to the latest upstream revision.  Use
					"YYMMDDhhmmss" to target a specific
					snapshot, or "latest" to restore the
					state from the most recent snapshot.
					This is the recommended way to recover
					from a plugin update that caused
					problems.
					(Default: "")

ATTRS                                                *dpp-ext-installer-attrs*

                                       *dpp-ext-installer-attr-installerBuild*
installerBuild  (String)
		Specify the build script.
		This command is executed by 'shell' from the plugin
		runtimepath.
		NOTE: It must be single command.
		NOTE: In previous versions of dpp, build could also be of
		type dictionary, but that is now deprecated.
		Please use |dpp-plugin-option-hook_post_update| instead.

                             *dpp-ext-installer-attr-installerMaxInactiveDays*
installerMaxInactiveDays (Number)
		Specify the maximum number of days since the previous commit
		that is allowed when updating a plugin.
		If the number of days between the previous commit and the new
		commit exceeds this value, a warning will be displayed
		indicating that the plugin has been inactive for an extended
		period before this update.
		If you don't specify this value,
		|dpp-ext-installer-param-maxInactiveDays| is used.

                               *dpp-ext-installer-attr-installerMinCommitDays*
installerMinCommitDays (Number)
		Specify the minimum number of days that must have passed since
		the previous commit in order to update a plugin.
		If you don't specify the value,
		|dpp-ext-installer-param-minCommitDays| is used.

                                      *dpp-ext-installer-attr-installerFrozen*
installerFrozen (Bool)
		If set to v:true, dpp doesn't update it automatically. It is
		useful for outdated plugins that can no longer be updated.

------------------------------------------------------------------------------
PARAMS                                              *dpp-ext-installer-params*

                                           *dpp-ext-installer-param-checkDiff*
checkDiff
		It checks plugins documentation diff when updated plugins.
		NOTE: It outputs the diff to new buffer in the background.

		Defaults: v:false

                                         *dpp-ext-installer-param-logFilePath*
logFilePath
		The log file output path.

		Defaults: ""

                                        *dpp-ext-installer-param-maxProcesses*
maxProcesses
		The max number of processes used for update.

		Defaults: 5

                                     *dpp-ext-installer-param-maxInactiveDays*
maxInactiveDays
		Specify the maximum number of days since the previous commit
		that is allowed when updating a plugin.

		Defaults: 180

                                       *dpp-ext-installer-param-minCommitDays*
minCommitDays
		Specify the minimum number of days that must have passed since
		the previous commit in order to update a plugin.

		If it is greater than 0, |dpp-ext-installer| will check the
		days between the previous and current commit when performing
		plugin updates.

		On |dpp-ext-installer-action-install| or
		|dpp-ext-installer-action-update|:
			If the interval is less than it, only a warning is
			shown, but the plugin is updated anyway.

		On |dpp-ext-installer-action-checkNotUpdated|:
			If the interval is less than it, a warning is
			displayed and the update will be **skipped**.

		It helps you avoid updating plugins too frequently (e.g.
		following experimental or fast-moving branches).

		Defaults: 0

                                                *dpp-ext-installer-param-wait*
wait
		The wait time(ms) after one plugin update.
		It is useful for restricted network environment.

		Defaults: 0

ROLLBACK                                          *dpp-ext-installer-rollback*

Before every |dpp-ext-installer-action-install|,
|dpp-ext-installer-action-reinstall|, or |dpp-ext-installer-action-update|
actions, `dpp-ext-installer` automatically saves the current revision of
every managed plugin to a snapshot file:

	{base-path}/{name}/rollbacks/{YYMMDDhhmmss}/rollback.json

Two snapshots are written each time:

- A timestamped directory (e.g. "260510153045") that permanently records the
  state at that exact moment.
- The special directory "latest", which is overwritten on every operation and
  always contains the most recent pre-operation snapshot.

Rollback is designed for local recovery: if a plugin update introduces a
regression you can immediately restore the previous state without having to
remember which revision was installed before.

To restore all plugins to the state before the last update:
>vim
	call dpp#async_ext_action('installer', 'update',
	\ #{ rollback: 'latest' })
<

To restore to a specific earlier snapshot:
>vim
	call dpp#async_ext_action('installer', 'update',
	\ #{ rollback: '260510120000' })
<

To restore only specific plugins to the previous state:
>vim
	call dpp#async_ext_action('installer', 'update',
	\ #{ names: ['dpp.vim'], rollback: 'latest' })
<

==============================================================================
EXAMPLES                                          *dpp-ext-installer-examples*
>vim
	" Install plugins
	call dpp#async_ext_action('installer', 'install')

	" Update plugins
	call dpp#async_ext_action('installer', 'update')

	" Update plugins with specify rollback revisions
	call dpp#async_ext_action('installer', 'update',
	\ #{ rollback: 'latest' })

	" Update dpp.vim
	call dpp#async_ext_action('installer', 'update',
	\ #{ names: ['dpp.vim'] })

	" Check not updated plugins
	call dpp#async_ext_action('installer', 'checkNotUpdated')

	" Get not installed plugins
	echo dpp#sync_ext_action('installer', 'getNotInstalled')
<

                                                     *dpp-ext-installer-faq-1*
Q: The installation/update is slow.

A: You can change |dpp-ext-installer-param-maxProcesses| value.

                                                     *dpp-ext-installer-faq-2*
Q: I want to update from shell.

A:

For neovim: >sh
	$ nvim --headless +'autocmd User Dpp:ext:installer:updateDone qall' \
	+'call dpp#async_ext_action("installer", "update")'
<

For Vim: >sh
	$ vim -c 'autocmd User Dpp:ext:installer:updateDone qall' \
	-c 'call dpp#async_ext_action("installer", "update")' \
	-Nu ~/.config/vim/vimrc -U NONE -i NONE -V1 -e -s
<

                                                     *dpp-ext-installer-faq-3*
Q: I want to quit Vim after updated plugins.

A: >vim
	autocmd User Dpp:ext:installer:updateDone quit
<

                                                     *dpp-ext-installer-faq-4*
Q: I want to get not updated plugins by Ddu.

A: >vim
	call ddu#start(#{
	    \   sources: [
	    \     #{
	    \       name: 'dpp',
	    \       params: #{
	    \         names: dpp#sync_ext_action('installer', 'getNotUpdated')
	    \                ->map({_, val -> val.name}),
	    \       },
	    \     },
	    \   ],
	    \ })
<
                                                     *dpp-ext-installer-faq-5*
Q: I want to update failed plugins.

A: >vim
	call dpp#async_ext_action('installer', 'update',
	    \ #{ names: dpp#sync_ext_action('installer', 'getFailed')
	    \ ->map({ _, val -> val.name }) })
<

                                                     *dpp-ext-installer-faq-6*
Q: Why doesn't "dpp-ext-installer" support lockfiles?

A: "dpp-ext-installer" mainly focuses on local plugin updates and recovery.
It prioritizes rollback support so users can restore previously installed
plugin revisions after an update causes problems.

Unlike plugin managers that are designed around Git repositories only,
|dpp.vim| uses extensible protocols and can support non-Git plugin sources.
Because plugin sources are not limited to a single backend, a generic
lockfile format is less straightforward.

For this reason, lockfiles are not currently supported.
They may be reconsidered in the future if there is clear demand.

                                                     *dpp-ext-installer-faq-7*
Q: What is the difference between rollback and lockfiles?

A: Rollback restores previously installed plugin revisions after an update.
Lockfiles are mainly used to reproduce the same plugin state across multiple
environments.

"dpp-ext-installer" currently prioritizes rollback for local recovery after
plugin updates.

                                                     *dpp-ext-installer-faq-8*
Q: I want to delay plugin updates as security feature.

A: You can specify |dpp-ext-installer-param-minCommitDays| and you can check
updates by |dpp-ext-installer-action-checkNotUpdated|.
